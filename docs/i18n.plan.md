# i18n

## Goal

Add Russian without making the site architecture clever.

The site is small, so the best approach is:

- keep English as the default locale at root paths
- put Russian under the top-level `/ru` prefix
- localize by page and by data file, not by sprinkling translation lookups
  across components

Target URLs:

- `/` for English home
- `/cv` for English CV
- `/ru` for Russian home
- `/ru/cv` for Russian CV

## Routing

Use locale as the first path segment for non-default locales.

That means:

- `src/pages/index.mdx` -> `/`
- `src/pages/cv.astro` -> `/cv`
- `src/pages/ru/index.mdx` -> `/ru`
- `src/pages/ru/cv.astro` -> `/ru/cv`

This matches the shape discussed earlier and keeps routing consistent across the
site.

Astro config should eventually express:

- locales: `["en", "ru"]`
- default locale: `"en"`
- default locale not prefixed

## Scope Split

There are two different i18n problems here:

- route i18n
- content i18n

They should stay separate.

Route i18n decides whether the page lives at `/cv` or `/ru/cv`.

Content i18n decides which data and strings the page renders.

## CV Strategy

Treat each CV translation as a complete document, not as a patch over English.

That means:

- `cv.yaml` is the English source of truth
- `cv.ru.yaml` is the Russian source of truth
- each file should independently satisfy the same `CV` TypeScript shape
- each file should include localized CV metadata such as a dedicated page title
  field, for example `head.cvTitle`

This is better than field-level translation maps because the CV is long-form
content:

- summaries are markdown
- some phrasing is naturally rewritten, not translated literally
- entries may eventually diverge by locale

## `cv.data.ts`

`cv.data.ts` should stop exporting a single default CV.

Instead, it should export a locale-aware API, for example:

```ts
export type Locale = "en" | "ru"

export const cvByLocale: Record<Locale, CV> = {
  en: parseCV(...),
  ru: parseCV(...),
}

export const getCV = (locale: Locale): CV => cvByLocale[locale]
```

This keeps the data static at build time while making page rendering
locale-aware.

The important distinction is:

- not runtime-fetched
- still statically imported
- selected by locale during page render

So the CV does not become "dynamic" in an undesirable sense. It only becomes
locale-indexed.

## `cv.astro`

Do not make one page component infer too much from global state.

A simple shape is:

- extract the shared CV markup into a reusable component, for example
  `src/app/cv/CVPage.astro`
- keep thin route entrypoints:
  - `src/pages/cv.astro`
  - `src/pages/ru/cv.astro`

Each route entrypoint should:

- choose the locale explicitly
- load the matching CV via `getCV(locale)`
- pass localized UI copy needed by the page

This is preferable to duplicating the full page markup in two route files.

## Page Strings

The CV page has two kinds of text:

- data content from YAML
- UI labels hardcoded in `cv.astro`

The page title should move into the CV data schema instead of staying in page
copy.

That means `cv.yaml` and `cv.ru.yaml` should both carry a localized field such
as `head.cvTitle`.

The remaining hardcoded labels currently are:

- `Summary`
- `Experience`
- `Education`
- `Honors & awards`
- icon alt text such as `email`, `linkedin`, `website`, `github`
- date word `Present` from `fmtExpTime()`

These should not stay inline in the page if Russian is added.

A minimal approach is a tiny locale dictionary dedicated to the CV page, for
example:

```ts
export const cvPageCopy = {
  en: { ... },
  ru: { ... },
} as const
```

This is enough. A global translation framework would be overkill for the current
site size.

## Date Formatting

`fmtDate()` and `fmtExpTime()` are currently English-only.

They will need locale input, because Russian needs at least:

- month names in Russian
- a Russian replacement for `Present`
- possibly different duration wording

Preferred direction:

- pass locale into formatting helpers
- keep formatting logic centralized in `src/app/cv/cv.ts`

Avoid:

- translating date fragments inside page templates
- hardcoding English date output in shared helpers

## Home Page Strategy

The homepage should be simpler than the CV.

Recommended shape:

- keep `src/pages/index.mdx` as English
- add `src/pages/ru/index.mdx` as Russian

This is better than trying to force one MDX file to branch by locale.

Reasons:

- the page is small
- MDX is content-first
- separate localized files are easier to edit and review

Articles can stay English-only for now.

That means article routes do not need Russian variants in this iteration.

## Locale Switcher

The first iteration should include a visible locale switcher on both `/` and
`/cv`.

Keep it simple:

- show a visible English/Russian switcher on the home page
- show a visible English/Russian switcher on the CV page
- switch to the equivalent localized route when one exists
- leave article links and article routes English-only

This should be added after the localized route structure is in place, but it is
part of the initial user-facing scope.

## Shared Components

Most existing CV components should remain locale-agnostic:

- `Org.astro`
- `Products.astro`
- `IconLink.astro`

They should render whatever localized data they receive.

The locale-aware decisions should stay near:

- route entrypoints
- data loaders
- localized copy modules
- formatting helpers

## Schema Discipline

The current direction only works cleanly if all locale variants share one
schema.

That includes any new localized CV title field added for routing and page
metadata.

Recommended rule:

- all locale files must match the same `CV` schema
- locale-specific optional fields should be modeled in TypeScript explicitly
- if `head.cvTitle` is added, it must exist in both `cv.yaml` and `cv.ru.yaml`

Avoid hidden per-locale shape drift, because it will leak complexity into every
consumer.

## Recommended Implementation Order

1. Add Astro i18n config for `en` and `ru`.
2. Add the shared localized CV title field to `cv.yaml` and `cv.ru.yaml`.
3. Refactor `cv.data.ts` to export locale-aware CV data.
4. Refactor `cv.ts` formatting helpers to accept locale.
5. Extract shared CV markup into a reusable component.
6. Add `src/pages/ru/cv.astro`.
7. Add `src/pages/ru/index.mdx`.
8. Add the visible locale switcher on `/` and `/cv`.

## Working Recommendation

If the goal is the smallest clean solution, I would proceed with these
assumptions:

- locale prefix only for non-default locale
- separate localized page files for `index` and `cv`
- English-only articles for now
- shared CV renderer component
- `getCV(locale)` from `cv.data.ts`
- one strict `CV` schema for all locales
- `head.cvTitle` stored in each CV locale file
- a visible locale switcher on `/` and `/cv`
- a tiny page-local copy dictionary for labels other than the CV title

That keeps i18n explicit, boring, and maintainable.
